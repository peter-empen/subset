com.osinka.subset

Subset

class Subset [T] extends Path

Subset is an abstract class to build subdocuments

Subset

  • encloses fields.
  • helps serialize/deserialize subdocuments to/from DBObject
  • participates in Query and Update creation
Nested documents

Since MongoDB documents may be nested, they sometimes make up a complex hierarchies with arrays of maps or maps of maps.

The fields created in Subset automatically get the scope of it, so that they have correct path in queries, update operators, etc.

The following is based upon a simple REPLable example (gist.github.com), showing a Blog post with comments (which resembles MongoDB's example here) Every comment has an author, a text and a number of votes.

case class Comment(by: String, votes: Int, text: String)
case class BlogPost(title: String, comments: List[Comment])

object Comment {
  val by = "by".fieldOf[String]
  val votes = "votes".fieldOf[Int]
  val text = "text".fieldOf[String]
}

object BlogPost {
  val title = "title".fieldOf[String]
  val comments = "comments".fieldOf[List[Comment]]

  object Comments extends Subset("comments") {
    val by = Comment.by.attach
    val votes = Comment.votes.attach
    val text = Comment.votes.attach
  }
}

BlogPost.Comments is a Subset. Fields by, votes and text are created inside it, use its scope (we call Field's attach method to avoid repetiting field names.

From now on, you may create queries and update operators using these fields and they will have long names in "dot notation", e.g. Comments.by === 10 will result in a query {"comments.by": 10}. Analogously, update operation comments.vote.inc(1) will result in {$inc: {"comments.vote": 1}}

Serialization

Subset provides a couple of helper methods for serializing an arbitrary number of fields under the subset's name and for deserializing a document from DBObject using the subset's name

apply takes an object and, assuming we have implicit ValueWriter in scope, creates a DBObjectLens storing the object into a fresh DBObject with subset's name. For example, Comments(new Comment("Joe", 2, "text")).get will result in {"by": "Joe", "votes": 2, "text": "text"} (you may find ValueWriter in the Gist)

Analogously, unapply makes use of ValueReader to extract an object. E.g. assuming we have implicit in scope (see the Gist for example), dbo match { case Comments(comment) => ... }

Querying and Updating

Along with simple queries all fields support, Subset lets you define much more advanced elemMatch queries and positional update.

Again, turning back to our example, how would you increment "votes" of a Joe's comment?

// Let's increment a Joe's comment vote
coll.update(BlogPost.Comments.by === "joe", BlogPost.Comments.updateMatch {_.votes inc 1})

(there is an alternative shown in Field)

It is also possible to find a BlogPost which has a comment made by "Joe" and having "votes" equal to 2:

// Find a blog post commented by Joe and the comment's vote is 2
coll.find(BlogPost.Comments.elemMatch {comment => comment.by === "joe" && comment.votes === 2}).iterator
Tracing

Since both Subset and Field are Path, you may check what "dot notation" Subset creates:

BlogPost.Comments.votes.at(1).in(BlogPost.Comments).longName must equal("comments.1.votes")
